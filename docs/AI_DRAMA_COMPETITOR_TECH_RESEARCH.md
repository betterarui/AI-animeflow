# AI 漫剧/短剧竞品技术调研报告

调研日期：2026-05-17  
用途：产品架构升级、竞品分析、技术路线规划  
口径：公开事实 + 合理架构推断，所有推断均单独标注

---

## 一页结论

现在优秀 AI 漫剧/AI 短剧产品，已经不是“输入一句话生成一个视频”的形态，而是在做一条更完整的生产流水线：

```text
长文本 / 创意 / IP
-> 剧本理解与改编
-> 角色 / 场景 / 道具 / 风格资产库
-> 分集 / 分场 / 分镜 / 台词 / 运镜
-> 关键帧 / 首尾帧 / 参考图
-> 图生视频 / 文生视频 / 参考生视频
-> 配音 / 口型 / 音乐 / 音效
-> 自动质检 / 重生成 / 人工选择
-> 时间线剪辑 / 字幕 / 封面 / 多平台导出
-> 发行 / 数据复盘 / 变现
```

最值得关注的能力有 5 个：

1. **长文本 IP 改编**：把小说、剧本、网文拆成分集、场景、角色关系和短剧节奏。
2. **资产圣经**：角色、场景、道具、服装、声线、风格图都要版本化，后续分镜持续引用。
3. **分镜级 DAG 工作流**：每个镜头独立生成、独立失败、独立重试，而不是整片一起重跑。
4. **多模型路由**：文本、图像、视频、TTS、口型、超分、审核分别调用不同模型。
5. **编辑与发行闭环**：优秀产品不止生成视频，还覆盖时间线编辑、团队协作、审核、分账和投放。

## 单品拆解

### 1. OiiOii

**产品定位**  
AI 动画/漫剧视频工具。公开资料将其描述为 AI Animation Agent Team，强调视频生成、风格库、多模型和批量渲染。

**核心功能链路**

- 创意、文本或图片输入。
- 生成故事、分镜、角色、场景。
- 调用图像/视频模型生成动画片段。
- 批量渲染后拼接导出。

**可证实信息**

- 官网有视频生成产品页。
- 工具资料提到 17+ 模型、批量并发渲染、多 Agent、托管/对话模式、风格库等。

**推断技术架构**

```text
用户输入
-> 多 Agent 调度器
-> 剧本 Agent / 角色 Agent / 分镜 Agent / 动画 Agent
-> 素材库与镜头任务队列
-> 图像模型 / 视频模型 / TTS 模型
-> 成片导出
```

**值得借鉴**

- 把“生成”包装成 Agent 团队，而不是暴露复杂参数。
- 支持并发渲染，有利于短剧镜头级量产。

**风险/短板**

- 核心模型、任务系统、角色一致性算法公开细节不足。

**来源**

- OiiOii 视频生成页：https://www.oiioii.ai/zh/features/video-generation
- OiiOii 工具资料：https://www.chooseai.net/tools/1123
- 证据等级：B

## 共性技术架构

### 1. 长文本理解层

解决问题：

- 小说/剧本太长，不能直接丢给视频模型。
- 需要提取主线、支线、人物关系、爽点、冲突、章节节奏。

常见实现：

```text
长文本分块
-> 章节摘要
-> 角色关系图
-> 场景和道具抽取
-> 分集大纲
-> 短剧节奏重写
```

代表产品：

- 小云雀
- 阅文漫剧助手
- 万兴剧厂
- 巨日禄
- 蛙蛙写作

### 2. 资产圣经层

解决问题：

- 连续剧最怕角色变脸、服装漂移、场景前后不一致。

常见资产：

- 角色卡
- 场景卡
- 道具卡
- 服装设定
- 声线设定
- 风格参考图
- 首帧/尾帧
- 模型、prompt、seed、版本记录

代表产品：

- FlovaAI：key element IDs 和资产组。
- 阅文漫剧助手：设定 Agent。
- Anijam：角色一致性。
- 360 纳米：空间、资产、视觉记忆。
- SkyReels：3D/空间中间表示。

### 3. 分镜导演层

解决问题：

- 视频模型只能生成短片段，短剧必须拆成镜头。

推荐数据结构：

```text
shot_id
episode_id
scene_id
characters
location
visual_description
dialogue
camera_movement
duration
first_frame
last_frame
reference_assets
video_model
generation_status
qc_status
```

代表产品：

- 万兴剧厂
- LibTV
- WorkRally
- SkyReels
- Anijam

### 4. 多模型路由层

解决问题：

- 没有一个模型能同时做好剧本、图像、视频、配音、口型、剪辑、审核。

常见路由：

| 任务 | 模型类型 |
| --- | --- |
| 剧本/分镜 | LLM / Agent |
| 角色图 | 图像生成 / 图像编辑 |
| 分镜图 | 图像生成 + 参考图控制 |
| 镜头视频 | I2V / T2V / R2V / 首尾帧模型 |
| 配音 | TTS / Voice clone |
| 口型 | Lipsync / Talking head |
| 后期 | FFmpeg / 时间线编辑器 / 超分 |
| 审核 | 内容安全 / 视觉一致性检测 |

代表产品：

- 爱奇艺纳逗 Pro
- 万兴剧厂
- WorkRally
- 小云雀
- SkyReels

### 5. 异步任务 / DAG 层

解决问题：

- 一集可能有几十个镜头，每个镜头生成时间不同、失败概率不同。
- 不能因为一个镜头失败就整集重跑。

推荐任务结构：

```text
Project
-> Episode
-> Scene
-> Shot
-> Asset generation task
-> Image generation task
-> Video generation task
-> Audio generation task
-> QC task
-> Export task
```

关键能力：

- 并发控制
- 失败重试
- 镜头级回滚
- 模型 fallback
- 成本统计
- 版本记录
- 任务恢复

代表产品：

- FlovaAI
- LibTV
- WorkRally
- Anijam
- 360 纳米

### 6. 画布 / 时间线层

解决问题：

- 用户不只要生成结果，还要选择、替换、重生成和剪辑。

常见形态：

- 无限画布
- 镜头列表
- 时间线
- 节点图
- 分镜看板
- 资产面板

代表产品：

- LibTV
- FlovaAI
- Anijam
- Doratoon
- WorkRally

### 7. 质检返工层

解决问题：

- AI 视频最常见问题是变脸、穿帮、动作错误、字幕水印、画风漂移、音画不同步。

常见检查项：

- 角色脸部一致性
- 服装一致性
- 场景一致性
- 镜头时长
- 画幅比例
- 是否有水印/字幕
- 是否多出角色
- 是否敏感内容
- 配音和字幕是否对齐
- 镜头之间是否连续

## 技术栈趋势

### 1. LLM / Agent 成为生产调度层

公开资料中，“Agent”不再只是聊天机器人，而是用来承担编剧、导演、美术、分镜、质检、剪辑等生产角色。

代表产品：

- 阅文漫剧助手：剧本 Agent、设定 Agent、分镜 Agent。
- WorkRally：专家级 Agent 和 Skill。
- OiiOii：AI Animation Agent Team。
- Anijam：AI Animation Agent。
- Seko：创编一体视频 Agent。

### 2. 视频模型走向多模态参考

短剧/漫剧不再只用文生视频，更多使用：

- 图生视频
- 首尾帧视频
- 参考图生视频
- 参考视频生视频
- 动作/声音驱动视频

代表底层模型或平台：

- Seedance 2.0
- Kling
- Vidu
- SkyReels
- HunyuanVideo
- Wan
- Veo
- Sora
- Runway

### 3. 3D/空间引擎解决连续性

单镜头视频模型很容易前后不一致，因此部分产品开始引入 3D 或空间记忆：

- SkyReels：3D/空间中间表示。
- 360 纳米：视频世界模型、四维影视空间。

### 4. 节点工作流适合专业生产

一键成片适合普通用户，节点和画布适合专业工作室。

代表产品：

- 哩布哩布 LibTV
- FlovaAI
- ComfyUI 生态

### 5. 云原生和异步任务是底座

大规模短剧生产一定需要：

- 对象存储
- GPU 任务队列
- 消息队列
- 工作流状态机
- CDN
- 模型网关
- 成本统计
- 权限与协作

腾讯云创意工坊相关资料明确提到公有云、云原生和统一模型接口，这类架构会成为行业标配。

### 6. IP、审核和发行成为平台壁垒

单纯生成工具会被模型供应商压缩，长期竞争力来自：

- 正版 IP 库
- 角色/艺人/数字资产库
- 审核与版权管理
- 工作室协作
- 分账和发行渠道

代表产品：

- 爱奇艺纳逗 Pro
- 阅文漫剧助手
- 漫剧工场
- 橙星梦工厂

## 对技术路线的借鉴建议

### 1. 先做资产圣经，再做视频生成

不要让每个镜头独立 prompt。应该先建立：

- 角色卡
- 场景卡
- 道具卡
- 服装卡
- 风格卡
- 声线卡
- 参考图版本

### 2. 把任务系统升级成 DAG

一集短剧不是一个任务，而是一组任务图：

```text
剧本
-> 资产
-> 分镜
-> 分镜图
-> 视频
-> 配音
-> 质检
-> 剪辑
-> 导出
```

每个节点都要支持：

- pending
- running
- completed
- failed
- retrying
- skipped
- superseded

### 3. 分镜是最核心的数据结构

每条分镜不仅要有 prompt，还要保存：

- 角色引用
- 场景引用
- 镜头描述
- 运镜
- 对白
- 时长
- 首帧
- 尾帧
- 模型参数
- 成本
- 质检结果

### 4. 模型路由要产品化

不要把模型写死。应允许：

- 按任务选择模型。
- 按成本选择模型。
- 按质量自动升级模型。
- 失败时自动 fallback。
- 对单个镜头手动换模型重跑。

### 5. 引入自动质检

第一阶段至少做：

- 角色是否变脸。
- 是否多出角色。
- 是否有水印/字幕。
- 画幅是否正确。
- 视频是否生成失败。
- 时长是否达标。
- 内容安全是否通过。

### 6. 编辑器不是附属功能

竞品正在从“一键生成”转向“可控迭代”。建议至少有：

- 镜头列表。
- 分镜图面板。
- 视频片段预览。
- 重生成按钮。
- 替换素材。
- 时间线导出。

### 7. 长期要做 IP/发行/协作闭环

只做生成工具容易被模型供应商替代。更长期的壁垒是：

- IP 管理
- 团队协作
- 版本历史
- 版权记录
- 审核记录
- 平台规格导出
- 数据复盘
